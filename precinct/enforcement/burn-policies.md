# Burn Policies — When -1 Is Acceptable

## The Fire Principle (火術)
Not all spend is waste. Strategic burns create value. But every burn must be justified.

## Approved Burns

### Category: Investment
- **New skill creation** — one-time cost, permanent capability
- **Repo bootstrapping** — front-loaded cost, ongoing returns
- **Agent training via delegation** — spend now, efficiency later
- **Infrastructure setup** — costs amortize over time

### Category: Maintenance
- **Morning briefings** — daily cost, daily value to human
- **Heartbeat checks** — low cost, high awareness
- **Memory maintenance** — preserves continuity, prevents re-learning

### Category: Emergency
- **Recovery from failure** — spend to prevent total loss
- **Human-requested urgent tasks** — override all budget limits
- **Security incidents** — no cost limit on protection

## Prohibited Burns

### Category: Waste
- ❌ Polling in tight loops
- ❌ Repeating failed operations without changing approach
- ❌ Loading full context when partial would suffice
- ❌ Spawning sub-agents for tasks that bash scripts can handle
- ❌ Path A operations when Path B exists

### Category: Vanity
- ❌ Verbose responses to internal/machine consumers
- ❌ Over-formatting for non-human audiences
- ❌ Generating documentation nobody will read
- ❌ Redundant confirmations and status updates

## Burn Approval Authority

| Tier | Approver | Examples |
|---|---|---|
| Tier 0 (< 1k tokens) | Self-approved | File reads, memory searches |
| Tier 1 (1k-10k tokens) | Agent discretion | Standard tasks, searches |
| Tier 2 (10k-50k tokens) | Fiesta (main) | Sub-agent spawns, complex tasks |
| Tier 3 (> 50k tokens) | Human approval | Multi-agent orchestration, new projects |
